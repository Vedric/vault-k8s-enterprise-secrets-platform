# Project Rules

## Identity
- NEVER mention Claude, Anthropic, AI, IA, LLM, or any AI-related terms in any output
- No "Generated by", "Co-authored-by", "AI-assisted", or similar attribution anywhere
- All output must appear as written by a senior engineer

## Language
- ALL output must be in English without exception
- This includes: code, comments, documentation, commit messages, PR descriptions, branch names, CHANGELOG, README, error messages, log messages, variable names, function names, constants, file names, folder names, and any text visible in the codebase
- No French, no other language — English only, everywhere, always

## Git Practices
- Commit messages follow Conventional Commits (feat:, fix:, chore:, docs:, refactor:, test:, ci:)
- Keep commits atomic — one logical change per commit
- Write meaningful commit messages explaining WHY, not just WHAT
- Never commit secrets, tokens, credentials, or .env files
- Always check for sensitive data before any git operation

## Code Quality
- Follow DRY, KISS, SOLID principles
- Add meaningful inline comments only where logic is non-obvious
- No commented-out code left in files
- Proper error handling — never silently swallow errors
- Use descriptive variable and function names
- Type hints in Python, proper typing in TypeScript

## Docker & Containers
- Always pin base image versions (no :latest)
- Multi-stage builds when applicable
- Run as non-root user
- Minimize image layers and final image size
- Include HEALTHCHECK instructions
- Labels following OCI image spec (org.opencontainers.image.*)

## CI/CD
- All pipelines must be idempotent
- Use pipeline variables for environment-specific values, never hardcode
- Cache dependencies where possible
- Fail fast — linting and static analysis first
- Security scanning (Trivy, SAST) in every pipeline

## Infrastructure as Code
- Terraform: use modules, remote state, lock files
- Always plan before apply
- Tag all resources consistently
- Use variables and locals, never hardcode values
- Document module inputs/outputs

## Security
- Zero critical/high vulnerabilities policy
- Never expose secrets in logs, outputs, or error messages
- Use least-privilege principle for all IAM/RBAC
- Scan images and dependencies before pushing

## Documentation
- README.md for every project with: purpose, prerequisites, quickstart, architecture
- Document non-obvious decisions in code comments or ADRs
- Keep docs up to date with code changes

## Shell / Bash
- Use shellcheck-compliant scripts
- set -euo pipefail at the top of every script
- Quote all variables
- Use meaningful exit codes
- Add usage/help functions for scripts with arguments
